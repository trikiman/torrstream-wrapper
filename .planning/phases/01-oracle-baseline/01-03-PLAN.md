# Phase 1 Plan 03: Reboot validation + smoke + credentials handoff

**Goal:** Prove the Oracle stack survives a reboot and serves the empty wrapper, then record what Phase 2 needs (credentials, URLs, webhook shape).

## Tasks

### Task 3.1 ã Reboot cycle

```bash
# Record "before" state
ssh ... "systemctl is-active torrserver flask-wrapper caddy"
ssh ... "ss -lntp | grep -E ':(80|443|8090|5000) ' "
ssh ... "swapon --show"

# Reboot
ssh ... "sudo reboot"

# Wait up to 90s for SSH to come back
for i in $(seq 1 18); do sleep 5; if ssh -o ConnectTimeout=3 ... "echo up" 2>/dev/null; then break; fi; done

# Re-check
ssh ... "systemctl is-active torrserver flask-wrapper caddy"
ssh ... "ss -lntp | grep -E ':(80|443|8090|5000) ' "
ssh ... "swapon --show"
```

All three services must come back `active`. Swap persists. Ports bind.

### Task 3.4 ã Decide auto-deploy webhook shape for Phase 2

On AWS, the wrapper has a `/api/github-webhook` endpoint that HMAC-verifies a POST from GitHub and runs `git pull --ff-only` + `systemctl restart flask-wrapper`.

For Phase 2 MIGR-03, decide:
- **D7 ã secret source**: Re-use the existing AWS webhook secret (from `/etc/torrstream/github-webhook.env` on AWS) so the same GitHub webhook URL hits both hosts during the overlap window, or generate a new secret just for Oracle?

Recommendation (default): **generate a new Oracle secret**, add it as a second webhook in GitHub repo settings. Keeps environments isolated and the DNS cutover becomes "remove AWS webhook, leave Oracle webhook".

- **D8 ã webhook URL during staging**: Does the Oracle webhook listen at `http://158.101.214.234/api/github-webhook` (IP) or do we need a staging DNS record first? Recommendation: **IP during staging** (GitHub webhooks can target raw IPs over HTTP).
