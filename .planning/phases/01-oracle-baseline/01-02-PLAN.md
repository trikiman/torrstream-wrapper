# Phase 1 Plan 02: Install TorrServer + Flask wrapper + Caddy

**Goal:** All three services installed and running on `vless-x86-2` in a staging/empty state. No user data, no production domain, but `curl http://158.101.214.234/api/status` via Caddy returns 200 JSON from the wrapper.

## Tasks

### Task 2.1 — Install TorrServer binary

```bash
# Latest release pin — check github.com/YouROK/TorrServer/releases. As of 2026-05 use MatriX.139 or newer.
TS_VER=MatriX.139
TS_ARCH=linux-amd64
sudo curl -fL -o /usr/local/bin/TorrServer \
  "https://github.com/YouROK/TorrServer/releases/download/${TS_VER}/TorrServer-${TS_ARCH}"
sudo chmod +x /usr/local/bin/TorrServer

# Generate BasicAuth credentials file
# accs.db format: plain text, one `username:password_hash` per line, bcrypt-hashed.
# TorrServer accepts plain text too when --httpauth is set and accs.db has `user pass\n` plaintext.
# Safer: generate bcrypt hash via python.
TS_USER=torrstream
TS_PASS=$(openssl rand -hex 16)
python3 -c "
import bcrypt, sys
u='$TS_USER'; p='$TS_PASS'.encode()
h=bcrypt.hashpw(p, bcrypt.gensalt()).decode()
print(f'{u}:{h}')
" | sudo tee /var/lib/torrserver/accs.db >/dev/null
sudo chown ubuntu:ubuntu /var/lib/torrserver/accs.db
sudo chmod 600 /var/lib/torrserver/accs.db

# Persist creds for the user (and for Plan 02 of this phase to consume)
sudo tee /etc/torrstream/torrserver.env <<EOF
TORRSERVER_USER=$TS_USER
TORRSERVER_PASS=$TS_PASS
EOF
sudo chmod 600 /etc/torrstream/torrserver.env
```

Note: TorrServer's exact auth flag syntax varies by release. Verify with `/usr/local/bin/TorrServer --help | grep -i auth` and adjust the systemd unit `ExecStart` if the flag name differs (e.g., `--httpauth` vs `--auth`).

### Task 2.2 — TorrServer systemd unit

```bash
sudo tee /etc/systemd/system/torrserver.service <<'EOF'
[Unit]
Description=TorrServer
After=network-online.target
Wants=network-online.target

[Service]
Type=simple
User=ubuntu
Group=ubuntu
WorkingDirectory=/var/lib/torrserver
ExecStart=/usr/local/bin/TorrServer --port 8090 --path /var/lib/torrserver --httpauth
Restart=on-failure
RestartSec=10
StandardOutput=append:/var/log/torrstream/torrserver.log
StandardError=append:/var/log/torrstream/torrserver.log

[Install]
WantedBy=multi-user.target
EOF

sudo systemctl daemon-reload
sudo systemctl enable torrserver
sudo systemctl start torrserver
sleep 3
sudo systemctl status torrserver --no-pager
```

**Verify:**
- `curl -s -o /dev/null -w '%{http_code}\n' http://localhost:8090/` prints `401` (auth challenge — auth working)
- `curl -s -u "$TS_USER:$TS_PASS" http://localhost:8090/echo` returns something (TorrServer echoes version)
- Externally: `curl http://158.101.214.234:8090/` → 401

### Task 2.4 — Flask wrapper systemd unit

```bash
sudo tee /etc/systemd/system/flask-wrapper.service <<'EOF'
[Unit]
Description=TorrStream Flask Wrapper
After=network-online.target torrserver.service
Wants=network-online.target

[Service]
Type=simple
User=ubuntu
Group=ubuntu
WorkingDirectory=/opt/torrstream/app
EnvironmentFile=/etc/torrstream/torrserver.env
Environment=TORRSERVER=http://127.0.0.1:8090
Environment=FLASK_RUN_PORT=5000
ExecStart=/opt/torrstream/venv/bin/python /opt/torrstream/app/app.py
Restart=on-failure
RestartSec=10
StandardOutput=append:/var/log/torrstream/flask.log
StandardError=append:/var/log/torrstream/flask.log

[Install]
WantedBy=multi-user.target
EOF

sudo systemctl daemon-reload
sudo systemctl enable flask-wrapper
sudo systemctl start flask-wrapper
sleep 3
sudo systemctl status flask-wrapper --no-pager
```

**Pass-through note:** the repo's `app.py` expects `TORRSERVER_AUTH` as a tuple if auth is on. Check `app.py`'s auth handling:
- If the code uses `TORRSERVER_USER` + `TORRSERVER_PASS` env vars via `requests.get(auth=TORRSERVER_AUTH)`, the unit above works.
- If it uses `TORRSERVER_AUTH="user:pass"`, adjust the env line before starting.
- If auth support is missing from `app.py` entirely, this is a code change that belongs in Phase 2 plan 02-02 (MIGR-03 covers wrapper code deployment). For Phase 1 staging, TorrServer on localhost with auth is fine; the wrapper talking to it just needs to send BasicAuth on its requests.

### Task 2.5 — Install Caddy + minimal Caddyfile

```bash
sudo DEBIAN_FRONTEND=noninteractive apt-get install -y caddy

sudo tee /etc/caddy/Caddyfile <<'EOF'
{
  auto_https off
}

:80 {
  reverse_proxy 127.0.0.1:5000
  encode gzip
  log {
    output file /var/log/torrstream/caddy-access.log
  }
}
EOF

sudo systemctl enable caddy
sudo systemctl restart caddy
sleep 3
sudo systemctl status caddy --no-pager
```

**Verify end-to-end:**
- `curl -s http://158.101.214.234/api/status` returns 200 JSON with `torrserver.ok: true`, `torrent_count: 0`, `wrapper.position_entries: 0`
- `curl -s http://158.101.214.234/manifest.json` returns 200 and contains `"name": "TorrStream"`
- `curl -s http://158.101.214.234/` returns the HTML shell

## Exit Criteria

- `systemctl is-active torrserver flask-wrapper caddy` all return `active`
- TorrServer on 8090 requires BasicAuth (returns 401 without creds)
- Flask wrapper on 5000 proxies TorrServer (includes BasicAuth in outbound requests) and exposes the API
- Caddy on 80 reverse-proxies the wrapper
- Credentials recorded in `/etc/torrstream/torrserver.env` and displayed to the user

## Rollback

- `sudo systemctl disable --now torrserver flask-wrapper caddy`
- `sudo rm /etc/systemd/system/{torrserver,flask-wrapper}.service /etc/caddy/Caddyfile`
- `sudo rm /usr/local/bin/TorrServer`
- `sudo apt-get purge -y caddy`
- Keep `/opt/torrstream` and `/var/lib/torrserver` if you want to retry; `rm -rf` them for full reset
