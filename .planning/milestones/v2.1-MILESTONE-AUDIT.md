## Post-Ship Bug Fixes (2026-05-13)

These were real production failures discovered when the user actually drove the
site/Lampa, not in synthetic verification. Each one was diagnosed live, fixed,
deployed via the GitHub auto-deploy webhook, and validated via MCP browser E2E.

### 1. iPad slow load (commit `8e15e9e`, before audit window)
- **Symptom:** Vidstack bundle loaded 14 sub-resources across 2 CDN hostnames; Russian DPI throttle made each TLS handshake serial → 1-3 minute load on iPad Safari.
- **Fix:** Self-host Vidstack bundle to `/static/vidstack/` (10 chunks + 2 CSS). Same-origin HTTP/2 multiplexing.
- **Verified:** 9/9 smoke pass after deploy.

### 2. Lampa "connect then drop" (commit `2746adf`)
- **Symptom:** Lampa connected to TorrServer then immediately dropped during piece buffering.
- **Root cause:** TorrServer `TorrentDisconnectTimeout=30s` was shorter than typical Lampa buffer pause.
- **Fix:** Bump to `300s`. Also added new HTTPS subdomain `ts.trikiman.shop` in Caddy to put TorrServer behind TLS for Lampa clients.
- **Verified:** 3 cycles of 45s idle on Big Buck Bunny — torrent stayed `Torrent working`, no eviction events in TorrServer journal.

### 3. Slow torrent download throughput (commit `26bd4ad`)
- **Symptom:** Prophet (5 seeders) downloaded at ~10 B/s; only 7 active peers despite 97 swarm peers.
- **Root cause:** Oracle Cloud Security List + Ubuntu iptables both blocked inbound TCP/UDP 22115 (BitTorrent peer port). TorrServer could only leech from peers who initiated outbound connections.
- **Fix:** Open port 22115 (TCP+UDP) on both Oracle Cloud SecList (via console MCP automation) and host iptables (persisted to `/etc/iptables/rules.v4`).
- **Verified:** Stress test (4 concurrent workers, 2 minutes) — peers grew 7→25, peak download 51.9 Mbit/s, CPU 21-75% (1 OCPU, 1 GB RAM, swap untouched), 0 errors over 158 byte-range requests.

### 4. tv.trikiman.shop "infinite loop" on Play (commit `35e1042`)
- **Symptom:** Click any torrent → spinner forever. SW v5 era.
- **Root cause:** Vidstack does dynamic `import()` of provider chunks (`vidstack-video-*.js`, `vidstack-html-*.js`) on first playback. Original downloader script only captured static `import "..."` statements; missed 8 provider files + 6 chunks. 404s on those imports threw unhandled promise rejections that left the player permanently stuck.
- **Fix:** Rewrote `_download_vidstack.py` to capture both static and dynamic imports, with context-aware classification (sibling `./vidstack-X.js` is a provider when emitted from `providers/`, a chunk when from `chunks/`). Now mirrors **18 chunks + 8 providers**.
- **Verified:** MCP E2E — clicked Big Buck Bunny → video loaded in 4s, played at 54s saved position.

### 5. Site → Lampa position sync asymmetric (commit `f4abc94`)
- **Symptom:** "If u watch on Lampa u can continue on site, but if u on site u cant continue on Lampa cus time didnt sync."
- **Root cause:** Plugin's `tryResume` did one-shot seek with 10s polling window. Lampa restores its own stale position from internal `file_view` localStorage entry first, then plugin's seek raced to apply the wrapper position. On slow streams (Vidstack metadata not ready in 10s) plugin gave up, Lampa won.
- **Fix:** Extended polling to 60s. Skip seek if Lampa is already within 5s of saved position. Detect "fight-back" (currentTime drops back near 0 after our seek) and re-apply up to 3 times.
- **Verified:** Set `position=250` on server → opened BBB in Lampa → currentTime went 0 → 250 within 3 seconds and played from there. POST flushes every 5s confirm Lampa→site direction also still works.

### 6. Prophet/MKV "infinite spinner" (commit `cf1de03`)
- **Symptom:** Big Buck Bunny (.mp4) played fine. Prophet/Hail Mary (.mkv) showed `<media-player>` with `data-media-type="unknown"` and never assigned `<video>.src`. Stuck at spinner.
- **Root cause:** Vidstack does dynamic `import("https://cdn.jsdelivr.net/npm/media-captions@next/...")` even on plain MP4/MKV playback (it pre-loads the cue parser). Original downloader noped that import to `icons-noop.js` (empty module). When Vidstack tried to instantiate the cue renderer with the empty export, it threw `TypeError: t is not a constructor`. The error was inside an unhandled promise so the user only saw a spinner.
- **Fix:** Self-host the entire `media-captions@next/dist/` tree (5 files, ~52 KB) under `/static/vidstack/captions/`. Rewrite the jsdelivr URL inside our Vidstack chunks to point local. Removed the over-eager noop.
- **Verified:** MCP E2E — Prophet (.mkv) loaded with `data-media-type="video"`, `videoSrc` assigned, `duration=5772s` (96 min), no console errors.

## Other Hardening Shipped (commit `92d4637`)

- `playFile()` now has a 5s watchdog: if Vidstack fails to assign `<video>.src`, fall back to setting `video.src` directly. Chrome decodes MKV+H.264 natively via plain HTMLVideoElement.
- Service worker bumped to `v8`; precaches Vidstack provider/chunk files at install time so a webhook redeploy race never breaks first-click playback.
